# RECALL Roadmap

> Last updated: April 2026
> Current version: **0.8.1**

---

## Design Law

RECALL is a structured document language. Three concerns are always kept separate:

| Concern | Mechanism | Changes via |
|---|---|---|
| Structure | Core elements (verbs) | Compiler release |
| Visual | Theme layer (palette, fonts, style-block) | `.rcpy` copybook |
| Reuse | Component library | `.rcpy` files / npm |

This separation is the invariant. New releases add elements or capabilities — they
never break the contract between layers.

## v0.7 — Strict Mode ✅ Complete

**Goal:** Structured compilation errors. No silent failures. Every diagnostic is
a stable, queryable code with message, reason, and hint.

**What shipped:**
- `src/diagnostics/` — types, collector, renderer, codes (25 error codes: RCL-001–021 + RCL-W01–W05)
- Parser location tracking — `loc` on every AST node (line/col/source)
- Type checker (`src/typechecker/index.ts`) — symbol table, structural, statement, component passes
- `recall check` — standalone type check without compiling
- `--strict` — warnings promoted to errors
- `--format json` — machine-readable diagnostics with `source` and `caret` fields
- `docs/ERROR-REFERENCE.md` — full per-code reference

All 25 diagnostic codes are now active. See `docs/STRICT-MODE-SPEC.md` for the full reference.

## v0.8 — AI-First Tooling + Language Evolution ✅ Complete

### AI-First CLI Surface

Five new commands that make the compiler its own documentation:

| Command | What it does |
|---|---|
| `recall schema [--json]` | Live element + PIC type registry — queries `BUILT_IN_ELEMENTS` directly |
| `recall explain <code> [--json]` | Per-code reference — queries `CODES` in `codes.ts` |
| `recall check --inspect` | Shows every DATA symbol generated by LOAD FROM before type-checking |
| `recall check --quiet` | No output — exit code only (0/1/2) for CI scripting |
| `recall stats <file> [--json]` | Orientation snapshot: field counts, groups, sections, warnings |
| `recall history <file> [--commits n]` | Git-aware DATA field diff between commits |
| `recall fix <file> [--dry-run\|--yes]` | Auto-apply safe fixes: PIC resize, AUTHOR, empty sections |

The design principle: the compiler is the documentation. Drift between docs and
implementation is structurally impossible because the CLI queries live source.

### Language Evolution (C#-Inspired, COBOL-Disciplined)

Three language features modernising the data layer while maintaining COBOL discipline:

**VALUE BLOCK / END VALUE** — multi-line string literals, auto-sized PIC:
```cobol
01 BODY-TEXT PIC X VALUE BLOCK.
   First line of content.
   Second line of content.
END VALUE.
```

**RECORD types** — reusable group shapes declared once, expanded anywhere:
```cobol
RECORD CARD-SHAPE WITH FIELDS
   CARD-TITLE  PIC X(60)
   CARD-BODY   PIC X(200).

01 CARDS.
   05 CARD-1 LIKE CARD-SHAPE.
   05 CARD-2 LIKE CARD-SHAPE.
```

**REQUIRED on ACCEPTS** — explicit contract on component parameters:
```cobol
ACCEPTS PAGE-TITLE REQUIRED, SUBTITLE, BADGE.
```
RCL-017 fires when a REQUIRED parameter is not provided at the call site.

### Intent Layer — COMMENT Clause

Additive metadata on DATA fields for AI tooling:
```cobol
01 FETCH-SCORE PIC X(10) VALUE "2978"
   COMMENT "CAL fetch score — higher is better; cascade header uses this".
```

COMMENT clauses surface in `recall check --inspect` and `recall check --format json`
under `fieldIntent`. No effect on compiled output.

## Design Observations — Toward 1.0

Friction points identified while building with RECALL. They inform what 1.0 needs
to harden.

---

### 1. `USING` vs `WITH DATA` vs `WITH PARAM` — three binding mechanisms

- `DISPLAY NAVIGATION USING NAV-ITEMS` — group reference for built-in elements
- `DISPLAY MY-COMP WITH DATA FIELD1, FIELD2` — DATA DIVISION binding for components
- `DISPLAY MY-COMP WITH FIELD "literal"` — literal override

The surface area is small but real. 1.0 should document the decision rule clearly
or consider unifying the mechanisms.

---

### 2. SPLIT layout requires two sub-SECTION children — not enforced

A SECTION with `WITH LAYOUT SPLIT` expects exactly two child SECTION elements.
If written flat, it compiles and renders with empty columns. Silent failure.

**1.0 fix:** warn or error when LAYOUT SPLIT has no child SECTIONs.

---

### 3. Palette key period bug — silent failure

```cobol
COLOR-BG.  "#080a10".   ← wrong: key stored as "COLOR-BG." — lookup fails silently
COLOR-BG   "#080a10".   ← correct
```

**1.0 fix:** palette parser should reject keys with trailing periods.

---

### 4. VALUE string content must be opaque to preprocessors

Multi-line VALUE strings containing RECALL syntax confused both the preprocessor
pipeline and the parser. Fixed in v0.7.x. The principle: content inside a VALUE
string is never RECALL source — it must be treated as opaque at every stage.

---

### 5. Unimplemented error codes (gap from v0.7)

Seven codes defined but not yet enforced:
RCL-006, RCL-013, RCL-014, RCL-015, RCL-016, RCL-018, RCL-W05.

Full details and implementation notes in `docs/STRICT-MODE-SPEC.md`.
Closing this gap is a 1.0 prerequisite.

## v1.0 — Stable Language

**Goal:** Language specification frozen. Compiler is a stable runtime. All defined
error codes fire. Component ecosystem exists.

**Prerequisites:**
- ✅ All 25 error codes actively enforced (completed v0.8.0)
- ✅ LAYOUT SPLIT structural validation (completed v0.8.1)
- ✅ Palette key period error — RCL-022 (completed v0.8.1)
- ✅ `@semanticintent/recall-ui` — standard component library published (v0.1.2)

**What 1.0 means for RECALL:**
- The core element vocabulary is complete and stable
- The theme layer (palette / font / style-block) is the documented extension point
- Component libraries are the unit of community contribution
- Breaking changes require a major version
- Silent failures are errors

**What 1.0 ships with:**
- `@semanticintent/recall-compiler` — stable compiler, strict by default
- `@semanticintent/recall-ui` v0.1.2 — standard component library: nav, hero, stat-section, card-section, footer + dark theme
- recall.cormorantforaging.dev — documentation site (built in RECALL)
